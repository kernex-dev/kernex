# Design: Typed observation table

> Companion to [proposal.md](proposal.md). ADRs ratify the six locks named in the proposal and resolve drift surfaced during implementation.

## ADR-001: Schema shape (lock OBS-01)

### Decision

Migration `018_observations.sql` adds a regular SQLite table (not a virtual table):

```sql
CREATE TABLE IF NOT EXISTS observations (
    id           TEXT PRIMARY KEY,
    sender_id    TEXT NOT NULL,
    type         TEXT NOT NULL
                 CHECK (type IN (
                     'bugfix', 'decision', 'pattern', 'config',
                     'discovery', 'learning', 'architecture'
                 )),
    title        TEXT NOT NULL
                 CHECK (length(title) > 0),
    what         TEXT,
    why          TEXT,
    where_field  TEXT,
    learned      TEXT,
    created_at   TEXT NOT NULL,
    updated_at   TEXT NOT NULL,
    deleted_at   TEXT
);

CREATE INDEX IF NOT EXISTS idx_observations_active
    ON observations (sender_id, created_at DESC)
    WHERE deleted_at IS NULL;

CREATE INDEX IF NOT EXISTS idx_observations_type_active
    ON observations (sender_id, type, created_at DESC)
    WHERE deleted_at IS NULL;
```

### Rationale

- **`id` is a UUIDv4 string**, generated by the store at insert time. Matches the message-id discipline introduced with `get_message_by_id` in `kernex-memory 0.7.0`.
- **`where_field` renamed** from the operator-facing `where`: `WHERE` is a SQL keyword. The Rust struct uses `where_field` and the serde rename annotation produces JSON field `"where"` for the wire format.
- **Soft-delete column + partial indices** mirror the `facts` pattern from migration `017_soft_delete.sql`. Default reads filter `WHERE deleted_at IS NULL`; the indices cover that branch.
- **Two indices on `(sender_id, ...)`**: one for time-ordered scans, one for type-filtered scans. The `type_active` index accelerates `search_observations(..., kind: Some(_), ...)`.
- **CHECK constraints at the DB layer**: `length(title) > 0` and `type IN (<7 values>)`. The Rust enum prevents unknown types at the API layer; the DB CHECK is the second wall for any non-Rust consumer or schema drift.

### Consequence

- The column order is fixed at migration time. Reordering later requires a recreate-and-migrate; not planned.
- Adding a new `ObservationType` variant in Rust must be paired with a migration that extends the SQL CHECK constraint. A Rust-only addition silently rejects the new value at write time.

## ADR-002: Rust types (lock OBS-02)

### Decision

`crates/kernex-memory/src/observation.rs` exports three public types, all `#[non_exhaustive]`:

```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq, Serialize, Deserialize)]
#[serde(rename_all = "lowercase")]
#[non_exhaustive]
pub enum ObservationType {
    Bugfix, Decision, Pattern, Config, Discovery, Learning, Architecture,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
#[non_exhaustive]
pub struct SaveEntry {
    pub sender_id: String,
    #[serde(rename = "type")] pub kind: ObservationType,
    pub title: String,
    pub what: Option<String>,
    pub why: Option<String>,
    #[serde(rename = "where")] pub where_field: Option<String>,
    pub learned: Option<String>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
#[non_exhaustive]
pub struct Observation {
    pub id: String,
    pub sender_id: String,
    #[serde(rename = "type")] pub kind: ObservationType,
    pub title: String,
    pub what: Option<String>,
    pub why: Option<String>,
    #[serde(rename = "where")] pub where_field: Option<String>,
    pub learned: Option<String>,
    pub created_at: SystemTime,
    pub updated_at: SystemTime,
}
```

`ObservationType` also exposes `as_db_str()` / `from_db_str()` for the store-layer string round-trip. `SaveEntry::new(sender_id, kind, title)` is the minimal constructor; optional fields default to `None`.

### Rationale

- **`#[non_exhaustive]` on all three types** prevents downstream pattern-match breakage when fields are added later. Adding a new `ObservationType` variant is non-breaking on the Rust side; the migration extending the SQL CHECK is the breaking step.
- **Rust `kind` vs JSON `type`**: `type` is a Rust keyword. `r#type` compiles but reads as noise; the serde rename keeps both the Rust API and the JSON wire format idiomatic.
- **Rust `where_field` vs JSON `where`**: same reasoning. Single canonical Rust identifier; serde rename to the operator-friendly JSON key.
- **`created_at` / `updated_at` as `SystemTime`** match the typed-row discipline introduced in `kernex-memory 0.7.0` for `MessageRow` and `HistoryRow`.
- **No `Default` for `SaveEntry`**: title and `sender_id` are required; a `Default` impl would imply they are not. Operators construct explicitly or deserialize from JSON.

### Consequence

- External crates that match exhaustively on `ObservationType` need a wildcard arm to compile against the `#[non_exhaustive]` shape. This is the intended forward-compat seam.

## ADR-003: Separate `search_observations` trait method (lock OBS-03)

### Decision

`MemoryStore::search_observations(&self, query, sender_id, limit, since, kind) -> Result<Vec<Observation>, MemoryError>` is a NEW trait method, distinct from `search_messages`. It hits the `observations_fts` virtual table (with insert / update / delete triggers keeping the mirror in sync).

```sql
CREATE VIRTUAL TABLE IF NOT EXISTS observations_fts USING fts5(
    title, what, why, where_field, learned,
    content='observations',
    content_rowid='rowid'
);

CREATE TRIGGER IF NOT EXISTS observations_ai
    AFTER INSERT ON observations WHEN new.deleted_at IS NULL
BEGIN
    INSERT INTO observations_fts (rowid, title, what, why, where_field, learned)
    VALUES (new.rowid, new.title, new.what, new.why, new.where_field, new.learned);
END;

CREATE TRIGGER IF NOT EXISTS observations_au
    AFTER UPDATE ON observations
BEGIN
    INSERT INTO observations_fts (observations_fts, rowid, title, what, why, where_field, learned)
    VALUES ('delete', old.rowid, old.title, old.what, old.why, old.where_field, old.learned);
    INSERT INTO observations_fts (rowid, title, what, why, where_field, learned)
    SELECT new.rowid, new.title, new.what, new.why, new.where_field, new.learned
    WHERE new.deleted_at IS NULL;
END;

CREATE TRIGGER IF NOT EXISTS observations_ad
    AFTER DELETE ON observations
BEGIN
    INSERT INTO observations_fts (observations_fts, rowid, title, what, why, where_field, learned)
    VALUES ('delete', old.rowid, old.title, old.what, old.why, old.where_field, old.learned);
END;
```

### Rationale

- **Different row shapes.** `MessageRow` has `(id, conversation_id, role, content, timestamp)`; `Observation` has the structured-field shape. Folding both into one method would force a sum-type return or strip the typed surface back to tuples. Both regress the typed-row discipline ratified for messages in 0.7.0.
- **Different filter semantics.** Messages have no `type` column; `search_messages(..., kind: Some(_))` cannot apply at the trait surface, only at a post-fetch filter. Observations have a real `type` column; the filter pushes into SQL via the `idx_observations_type_active` index.
- **Different lifecycles.** Messages are auto-generated by every runtime exchange; observations are deliberate operator writes. Keeping the methods separate preserves the distinction at the API.
- **Downstream consumers can compose both surfaces.** A CLI / REPL surface that wants to query both messages and observations calls both methods and interleaves results at the consumer layer.

### Consequence

- Consumers issuing both queries pay two round trips for combined search. Acceptable given the existing cold-start bench measures `search_messages` at ~1.9 ms median on macOS aarch64 release; doubling that stays well under the 50 ms p95 informational target.
- The FTS5 mirror requires triggers to stay in sync with soft-delete state. The `observations_au` trigger removes a row from the FTS mirror whenever `deleted_at` becomes non-NULL, so search default-filtering of soft-deleted rows is automatic.

## ADR-004: 4-tuple `get_memory_stats` breaking change (lock OBS-04)

### Decision

`MemoryStore::get_memory_stats(sender_id: &str) -> Result<(i64, i64, i64, i64), MemoryError>` returns `(conversations, messages, observations, facts)`. The prior 3-tuple `(conversations, messages, facts)` is reshaped; the observation count joins at position 2.

The fact count continues to filter `WHERE deleted_at IS NULL` (matches the existing soft-delete discipline). The observation count uses the same discipline: `SELECT COUNT(*) FROM observations WHERE sender_id = ? AND deleted_at IS NULL`. Conversations and messages have no soft-delete column today; they are counted whole.

### Rationale

- **Bundling the semantic change with the trait additions** keeps the breaking-release count to one. Splitting into two minor versions (`0.8.0` = trait additions, `0.8.1` = stats reshape) imposes the migration cost twice on every downstream consumer.
- **`observations` is statistically distinct from `messages`.** Messages are generated by every runtime exchange (high cardinality); observations are deliberate operator writes (low cardinality but high value). Conflating them in a single count undercounts the annotation discipline operators care about.

### Consequence

- Every consumer of `get_memory_stats` must destructure four elements after the bump. The first consumer to migrate is `kernex-runtime/examples/full_stack.rs`; second is `kernex-agent`'s stats handler.

## ADR-005: Sandbox refusal stays at the consumer layer (lock OBS-04 corollary)

### Decision

`kernex-memory` has no knowledge of any sandbox. Any caller surface (CLI, REPL, HTTP) that needs to gate writes does so BEFORE calling `MemoryStore::save_observation`. On refusal, the consumer maps to its own error variant; the store layer never sees a sandbox failure.

### Rationale

- **`kernex-memory` is a pure store.** Adding sandbox awareness pollutes the layer with concerns it has no business owning.
- **Sandbox configuration is per-binary.** The store is shared by REPL, CLI, serve, and future surfaces; each surface decides its own write-gating policy.

### Consequence

- `MemoryStore::save_observation` never returns a sandbox-flavored error. Consumers depending on a distinct exit code (e.g. `kx mem save` exit 4 per the agent-side spec) must generate it pre-store.

## ADR-006: Forward-only migration (lock OBS-06)

### Decision

Migration `018_observations.sql` is non-reversible. If a deployment needs to discard the new table, the operator's path is to remove the local SQLite DB file (rebuilds clean on next open). No `019_revert_observations.sql` ships.

### Rationale

- **kx is a per-developer CLI with local SQLite databases**, not a multi-tenant server. The cost of a wrong migration is one deletion of one local DB.
- **The `_migrations` registry is authoritative.** Reapplying migration 018 against a DB that already has it is safe because of `IF NOT EXISTS` on every CREATE statement; the registry fast-path short-circuits before re-executing.

### Consequence

- Once shipped in `kernex-memory 0.8.0`, every consumer that opens an existing 0.7.x DB gets the new table on first open. No opt-out at the store layer.

## Trait surface diff

```rust
// Added by this change (5 methods on MemoryStore):

async fn save_observation(&self, entry: SaveEntry)
    -> Result<Observation, MemoryError>;

async fn get_observation_by_id(&self, id: &str)
    -> Result<Option<Observation>, MemoryError>;

async fn search_observations(
    &self,
    query: &str,
    sender_id: &str,
    limit: i64,
    since: Option<SystemTime>,
    kind: Option<ObservationType>,
) -> Result<Vec<Observation>, MemoryError>;

async fn soft_delete_observation(&self, id: &str)
    -> Result<bool, MemoryError>;

async fn list_soft_deleted_observations(&self, sender_id: &str)
    -> Result<Vec<Observation>, MemoryError>;

// Modified by this change (1 method, breaking):
async fn get_memory_stats(&self, sender_id: &str)
    -> Result<(i64, i64, i64, i64), MemoryError>;  // was (i64, i64, i64)
```

Inherent `Store` impls land in `crates/kernex-memory/src/store/observations.rs`; one private helper (`count_observations`) lives there to keep the `get_memory_stats` body small.

## Open questions deferred to future changes

1. **Multi-type filter on `search_observations`.** `kind: Vec<ObservationType>` would let a single search target multiple types. Deferred until a real operator ask surfaces; v1 ships `Option<_>` to match the existing single-`--type` flag on the agent-side spec.
2. **Structured `where_field`.** `WhereLocation { file: PathBuf, range: Option<(u32, u32)> }` would enable structured queries by file path or line range. Deferred; v1 is free text.
3. **Observation supersession links.** `link_observations(parent: &str, child: &str)` to track "this observation replaces that one". Deferred to a separate change once the primitive proves useful.
